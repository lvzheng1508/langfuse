# Langfuse 平滑升级策略分析

## 升级顺序：先升级 Schema，再升级代码

### ✅ 可行性分析

#### PostgreSQL 迁移兼容性

**organizations 表新增字段：**
1. `cloud_billing_cycle_anchor` - TIMESTAMP(3) DEFAULT CURRENT_TIMESTAMP, 可为 NULL
2. `cloud_billing_cycle_updated_at` - TIMESTAMP(3), 可为 NULL
3. `cloud_current_cycle_usage` - INTEGER, 可为 NULL
4. `cloud_free_tier_usage_threshold_state` - TEXT, 可为 NULL
5. `ai_features_enabled` - BOOLEAN NOT NULL DEFAULT false

**兼容性评估：**
- ✅ 所有字段都有默认值或可为 NULL
- ✅ 旧代码不会查询这些字段（除非明确 SELECT）
- ✅ Prisma 查询会自动处理缺失字段（返回 null 或默认值）
- ⚠️ 但当前代码在查询 organizations 时会 SELECT 所有字段，包括这些新字段
- ⚠️ 如果字段不存在，Prisma 查询会失败

**结论：** 
- 如果先升级 Schema，旧代码**可能**会失败，因为 Prisma Client 生成的类型期望这些字段存在
- 但如果旧代码不查询这些字段，应该可以工作

#### ClickHouse 迁移兼容性

**observations 表新增字段：**
1. `usage_pricing_tier_id` - Nullable(String)
2. `usage_pricing_tier_name` - Nullable(String)
3. `tool_definitions` - Map(String, String) DEFAULT map()
4. `tool_calls` - Array(String) DEFAULT []
5. `tool_call_names` - Array(String) DEFAULT []

**兼容性评估：**
- ✅ 所有字段都有默认值或可为 NULL
- ✅ 代码中有默认值处理：`usagePricingTierId ?? null`, `toolDefinitions ?? null`
- ⚠️ 但如果查询时 SELECT 这些字段而字段不存在，ClickHouse 会报错
- ⚠️ 代码中确实会查询这些字段（见 observations.ts）

**结论：**
- 如果先升级 Schema，旧代码查询这些字段时会失败
- 需要确保查询语句使用 `IF EXISTS` 或不在 SELECT 中包含这些字段

### ⚠️ 风险分析

#### 高风险场景

1. **Prisma Client 类型检查**
   - 旧代码使用旧版本的 Prisma Client，生成的类型不包含新字段
   - 但如果数据库有这些字段，Prisma 查询可能仍然成功（取决于查询方式）

2. **ClickHouse 查询失败**
   - 如果旧代码的 SQL 查询包含新字段名，而字段不存在，查询会失败
   - 代码中确实会查询这些字段

3. **代码依赖新字段**
   - 当前代码（v3.147.0）已经依赖这些字段
   - 如果先升级 Schema 但使用旧代码，可能会有问题

### ✅ 推荐的升级策略

#### 方案 1：先升级 Schema，再升级代码（需要验证）

**步骤：**
1. 备份数据库
2. 运行 PostgreSQL 迁移（只添加字段，不删除）
3. 运行 ClickHouse 迁移（只添加字段）
4. 验证旧代码是否仍能正常工作
5. 如果正常，升级代码
6. 如果异常，回滚 Schema

**优点：**
- Schema 变更先完成，降低风险
- 可以验证 Schema 变更是否正确

**缺点：**
- 旧代码可能因为查询新字段而失败
- 需要仔细测试

#### 方案 2：先升级代码，再升级 Schema（推荐）

**步骤：**
1. 备份数据库
2. 部署新代码（但代码会处理字段缺失的情况）
3. 运行数据库迁移
4. 验证功能正常

**优点：**
- 代码已经处理了字段缺失的情况（有默认值）
- 更安全

**缺点：**
- 需要确保代码能处理字段缺失

#### 方案 3：同时升级（最安全）

**步骤：**
1. 备份数据库
2. 维护窗口
3. 同时升级 Schema 和代码
4. 验证功能

**优点：**
- 最安全，避免不匹配问题

**缺点：**
- 需要维护窗口

### 🔍 关键发现

从代码分析来看：

2. **ClickHouse observations 字段**：代码会查询这些字段，如果字段不存在，ClickHouse 查询会失败

**结论：** 先升级 Schema 再升级代码**可能不可行**，因为当前代码（v3.147.0）已经依赖这些字段。

### 💡 建议

**推荐方案：同时升级 Schema 和代码**

1. 准备维护窗口
2. 备份数据库
3. 同时执行：
   - 运行数据库迁移
   - 部署新代码
4. 验证功能

**如果必须分步升级：**

1. 先升级代码到支持字段缺失的版本（如果有）
2. 然后升级 Schema
3. 最后升级到最新代码

但当前代码版本（v3.147.0）已经依赖这些字段，所以**必须同时升级**。

## 🔬 详细技术分析

### PostgreSQL Prisma 查询分析

**关键问题：** Prisma 在查询时会使用生成的 Prisma Client，该 Client 基于 schema.prisma 文件生成。

**如果先升级 Schema：**
1. 旧代码使用旧版本的 Prisma Client（基于旧 schema）
2. 旧 Prisma Client 生成的查询**不会**包含新字段
3. 但如果使用 `findMany()` 或 `findUnique()` 不指定 select，Prisma 会查询所有字段
4. 如果数据库有字段但 Prisma Client 不知道，查询**可能**会成功（取决于 Prisma 版本）

**实际情况：**
- Prisma 6.x 在查询时会根据 schema 生成 SQL
- 如果 schema 中没有字段，生成的 SQL 不会包含该字段
- 所以：**先升级 Schema，旧代码应该可以工作**（因为旧代码的 Prisma Client 不会查询新字段）

### ClickHouse 查询分析

**关键问题：** ClickHouse 查询是手写的 SQL，直接查询字段名。

**如果先升级 Schema：**
1. 旧代码的 SQL 查询**不包含**新字段（因为代码是旧版本的）
2. 所以查询应该可以成功
3. 但如果新代码已经部署，SQL 会包含新字段，查询会失败

**实际情况：**
- 当前代码（v3.147.0）的 SQL **包含**这些字段
- 如果先升级 Schema，然后部署新代码，应该可以工作
- 但如果先升级 Schema，旧代码仍在运行，旧代码的 SQL 不包含这些字段，应该也可以工作

### ✅ 最终结论

**先升级 Schema，再升级代码是可行的！**

**原因：**
1. **PostgreSQL**: 旧代码的 Prisma Client 不会查询新字段（因为 schema 中没有），所以不会出错
2. **ClickHouse**: 旧代码的 SQL 不包含新字段，所以不会出错
3. **新字段都有默认值或可为 NULL**，不会影响数据完整性

**但需要注意：**
- 确保旧代码版本不会查询这些字段
- 如果旧代码版本也会查询这些字段，则不可行
- 建议先测试一个非关键环境

### 📋 推荐的升级步骤

## 🎯 关键代码检查结果

### PostgreSQL organizations 表查询分析

**检查结果：**
- 大部分查询使用 `select` 明确指定字段，**不包含新字段**
- 但有一个关键查询在 `auth.ts` 的 session 回调中会查询 organization 的所有字段（通过 include）
- 这个查询会失败，如果字段不存在

**关键代码位置：**
- `web/src/server/auth.ts` - session 回调中查询 organization
- 使用 `include: { organization: true }` 会查询所有字段

**结论：**
- ⚠️ **如果先升级 Schema，旧代码会失败**（因为查询 organization 时会尝试读取新字段）
- 但如果旧代码版本不查询这些字段，应该可以工作

### ClickHouse observations 表查询分析

**检查结果：**
- 代码中明确 SELECT 了这些字段：
  - `o.usage_pricing_tier_id`
  - `o.usage_pricing_tier_name`
  - `length(mapKeys(o.tool_definitions))`
  - `length(o.tool_calls)`
- 如果字段不存在，ClickHouse 查询会**直接失败**

**结论：**
- ⚠️ **如果先升级 Schema，旧代码会失败**（因为 SQL 查询包含这些字段）

## ✅ 最终建议

### 方案：先升级 Schema，再升级代码（需要确认旧代码版本）

**可行性：** ⚠️ **有条件可行**

**前提条件：**

2. 如果旧代码版本也会查询这些字段，则**不可行**

**如果旧代码版本较旧（不查询新字段）：**
- ✅ 可以先升级 Schema
- ✅ 旧代码可以继续运行
- ✅ 然后升级代码

**如果旧代码版本较新（已查询新字段）：**
- ❌ 必须先升级代码，或同时升级
- ❌ 否则查询会失败

### 推荐方案：同时升级（最安全）

**步骤：**
1. 准备维护窗口（建议 30-60 分钟）
2. 备份数据库
3. 停止服务
4. 运行数据库迁移
5. 部署新代码
6. 启动服务
7. 验证功能

**优点：**
- 避免版本不匹配问题
- 最安全可靠

**缺点：**
- 需要维护窗口
- 服务会短暂中断

### 如果必须分步升级

**方案：先升级 Schema，再升级代码**

**前提：** 必须确认旧代码版本不查询新字段

**步骤：**
1. 检查当前运行的代码版本
2. 确认该版本是否查询新字段
3. 如果**不查询**，可以：
   - 先升级 Schema
   - 验证旧代码仍能运行
   - 再升级代码
4. 如果**查询**，必须同时升级

**验证方法：**
- 检查旧代码的 SQL 查询是否包含新字段名
- 检查 Prisma schema 是否包含新字段定义
